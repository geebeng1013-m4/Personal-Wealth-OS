# Personal Wealth OS 项目开发规范

## 1. 适用范围

- 本规范适用于 `personal-wealth-os/` 目录中的所有代码、配置与资源。
- 修改前先阅读相关模块及其调用方，遵循现有架构，不进行与任务无关的重构。
- 仓库中还有其他独立项目；除非任务明确要求，不要修改 `evolution-*`、`shared-assets/` 或根目录下的其他项目文件。
- 用户的本地改动可能尚未提交。不得覆盖、回退或删除与当前任务无关的改动。

## 2. 技术栈与常用命令

- 技术栈：Vite 5、原生 TypeScript、原生 DOM/CSS、Firebase、Lightweight Charts。
- TypeScript 使用 ES Modules，配置为 `strict: true`，目标运行环境为 ES2022。
- 在 `personal-wealth-os/` 目录运行：
  - `npm run dev`：启动本地开发服务器。
  - `npm run typecheck`：执行 TypeScript 类型检查。
  - `npm run build`：先进行类型检查，再生成生产构建。
  - `npm run preview`：预览生产构建。
- 项目同时存在 `package-lock.json` 和 `pnpm-lock.yaml`。除非任务明确涉及依赖，不要更新锁文件；需要安装依赖时，先确认本次任务采用的包管理器。

## 3. 模块职责

- `src/models.ts`：领域模型和共享类型。新增持久化字段时必须同步考虑默认值、迁移、导入导出和云端兼容性。
- `src/rules.ts`：财务计算和建议规则。函数应尽量保持纯函数，不在此模块读写 DOM、Local Storage 或 Firebase。
- `src/state.ts`：默认状态、本地持久化、版本迁移、快照及导入导出。所有持久化状态结构变更都必须经过这里处理。
- `src/firebase.ts`：身份验证与 Firestore 访问。Firebase 细节不得散落到 UI 或规则模块。
- `src/market.ts`：行情请求、缓存、格式化及风险指标。外部接口数据必须视为不可信输入。
- `src/ui.ts`：页面模板和事件绑定。模板生成与绑定函数应保持对应，避免只渲染控件却未绑定行为。
- `src/main.ts`：应用启动、认证流程、主题、导航和顶层状态协调。领域计算不要放入入口文件。
- `src/styles.css`：全局视觉样式与响应式规则。优先复用现有变量、布局和组件类。
- `public/sw.js`：PWA Service Worker 源文件。不要直接编辑构建产物 `dist/`；需要发布的缓存策略应在源文件中修改后重新构建。

## 4. TypeScript 与代码风格

- 保持严格类型，不使用 `any` 规避类型错误。无法立即确定的外部数据先使用 `unknown`，经过校验和收窄后再使用。
- 优先使用现有的接口、联合类型和辅助函数；领域概念应定义在 `models.ts`，不要在多个模块重复声明形状相近的类型。
- 使用具名函数表达业务意图。变量和函数使用 `camelCase`，类型和接口使用 `PascalCase`，常量使用项目现有风格。
- 保持函数职责单一。仅当抽取能消除真实重复或隔离复杂逻辑时才增加抽象。
- 不吞掉异常。面向用户的失败应显示可理解的状态，开发诊断应保留足够上下文，但不得记录敏感财务数据或认证令牌。
- 注释解释原因、约束或不明显的算法，不复述代码。
- 保持现有格式风格：双引号、分号和 ES Module `import`/`export`。

## 5. 状态与数据安全

- `WealthState.version` 是持久化结构的契约。新增、删除、重命名字段或改变字段语义时，必须提升版本并在 `migrateState` 中提供向后兼容迁移。
- 迁移必须保留旧数据。缺失字段使用明确默认值，不得因单个无效字段重置整个用户状态。
- 状态更新保持不可变语义，构造新的对象或数组后交给顶层 `setState`；不要在渲染过程中原地修改当前状态。
- 所有金额、比例、数量和日期输入都要校验。拒绝 `NaN`、`Infinity`、负数等不符合领域规则的值，并明确处理空输入。
- 货币单位必须写清楚。不得混算 MYR 与 USD；汇率换算必须显式，并保留原始交易金额。
- 金融计算不得依赖格式化后的字符串。内部使用数值计算，只在展示边界调用 `money`、`percent`、`formatPrice` 等格式化函数。
- 导入文件和旧版状态都属于不可信数据。解析后必须校验、迁移，再写入本地或云端。
- 会覆盖、清空、恢复或批量导入数据的操作必须要求明确确认，并尽可能先保存快照。

## 6. Firebase 与同步

- 未认证用户的数据必须按现有本地存储逻辑隔离；已认证数据必须以 `uid` 隔离，禁止跨用户读取或写入。
- 修改同步逻辑时要考虑首次登录、重定向登录、离线、重复订阅、退出登录、云端为空和本地/云端冲突。
- Firestore 写入失败不得导致本地数据丢失。UI 应反映失败或离线状态，不得伪装为同步成功。
- 订阅监听器必须在用户切换、退出或重新订阅时正确释放，避免重复回调和旧用户数据泄漏。
- 不在源码中提交密钥、服务账号、私有令牌或真实用户财务数据。客户端 Firebase 配置只能包含可公开的 Web 配置，并依赖 `firestore.rules` 实施访问控制。
- 修改数据访问模式时同步检查 `firestore.rules`，确保用户只能访问自己的文档。

## 7. 市场数据与财务建议

- 行情、基本面、历史价格和代理服务都可能超时、限流、返回空数据或改变字段。请求必须有错误处理，解析前校验结构。
- 缓存键必须包含影响结果的参数；缓存过期策略应明确。失败时可以展示陈旧数据，但必须标识其时间或状态，不能冒充实时行情。
- 并发请求应避免竞态：用户切换股票后，旧请求结果不得覆盖新选择的页面状态。
- 风险指标和盈亏计算要覆盖空序列、零成本、缺失基准、非交易日及长度不一致的数据。
- 自动建议应保持规则透明、语气克制，并明确属于信息或规划辅助。不要把估算收益、历史表现或规则输出描述为保证结果或个性化投资承诺。

## 8. UI、可访问性与安全

- 延续现有单页应用的信息架构和视觉语言。财务界面优先清晰、紧凑和易扫描，不添加与任务无关的营销式页面或装饰。
- 用户输入、CSV 内容、Ticker、备注及远端数据插入 HTML 前必须经过 `escapeHtml` 或等效的上下文安全处理。不要把未清洗字符串直接传给 `innerHTML`。
- 新增交互控件时同时实现默认、悬停、聚焦、禁用、加载、空数据和错误状态。
- 图标按钮要有 `aria-label` 或可见文本；表单控件要有对应标签；键盘可以完成主要操作。
- 保持移动端可用，避免固定宽度导致横向溢出。金额、长名称和错误信息必须能够换行或截断，不得遮挡其他控件。
- 颜色不是表达涨跌、风险或状态的唯一方式；同时提供文字、符号或结构提示。
- 修改主题相关样式时同时验证明暗主题，并确保文字、边框、图表和焦点状态有足够对比度。
- 不手工编辑 `dist/`。构建输出由 Vite 生成，不应作为源代码维护。

## 9. PWA 与离线行为

- 修改静态资源、入口文件或缓存策略时检查 Service Worker 的缓存版本和资源清单。
- Service Worker 更新不得长期保留不兼容的旧应用壳；必要时提升缓存版本并清理旧缓存。
- 离线模式下，已保存的本地财务数据仍应可访问；依赖网络的登录、同步和行情功能应明确显示不可用或缓存状态。
- 不缓存包含其他用户私有数据的响应，也不在共享缓存中保存认证信息。

## 10. 验证要求

- 每次代码修改后至少运行 `npm run typecheck`。
- 修改构建、PWA、入口、依赖或跨模块契约时运行 `npm run build`。
- 涉及 UI 时，在桌面和移动尺寸手动验证：导航、表单提交、空状态、错误状态、主题切换以及无溢出/遮挡。
- 涉及状态时验证：新用户默认状态、现有数据迁移、刷新后持久化、导入导出、快照恢复，以及登录/退出后的用户隔离。
- 涉及市场数据时验证：成功、超时、无数据、格式异常、快速切换 Ticker 和缓存过期。
- 涉及 Firebase 时验证：未登录、登录、退出、离线、权限拒绝和订阅清理，并复核 Firestore 规则。
- 当前项目没有自动化测试脚本。对复杂财务规则、迁移或解析逻辑，应优先补充可重复的测试；若本次无法补充，交付说明中必须列出未覆盖风险和人工验证结果。

## 11. 完成标准

- 需求已实现，且没有修改任务范围外的行为。
- `npm run typecheck` 通过；适用时 `npm run build` 通过。
- 没有新增 TypeScript 错误、浏览器控制台错误、明显可访问性问题或敏感信息泄漏。
- 持久化结构变更包含版本迁移，UI 变更包含响应式和错误状态，云端变更包含权限与失败路径检查。
- 交付说明简要列出修改文件、行为变化、验证命令以及任何剩余风险。
